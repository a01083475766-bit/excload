# 송장번호 변환 페이지 — 설계 문서

## 1. 개요

### 1.1 목적

- 주문 엑셀과 택배사에서 받은 송장(운송장) 번호 파일을 병합한 뒤, **쇼핑몰 송장 업로드용 양식**에 맞춰 출력하는 기능을 제공한다.
- 기존 **주문변환·물류변환** 페이지와 **레이아웃·카드·미리보기 구조를 최대한 동일**하게 유지하여 사용자 거부감을 줄인다.

### 1.2 범위에서 제외(해당 페이지)

- 텍스트 주문 입력 변환
- 스크린샷(캡처) 주문 변환

위 기능은 본 페이지에서 **제거**하고, 해당 UI 영역은 **두 번째 파일 업로드**로 대체한다.

---

## 2. UI 원칙

| 항목 | 방침 |
|------|------|
| 레이아웃 | 주문변환 등 기존 페이지와 동일한 구역(파일 선택 / 미리보기 / 하단 카드) 유지 |
| 파일 입력 | 기존 1개 업로드 + **동일 스타일 업로드 1개 추가** (총 2개) |
| 미리보기 영역 | 구조·역할 동일, **페이지에 맞는 안내 문구만** 조정 |
| 하단 카드 | 기존과 동일한 흐름(양식 등록 → 고정 입력 → 다운로드) 유지, 라벨만 송장/쇼핑몰 맥락에 맞게 조정 |

### 2.1 파일 업로드 영역 문구(권장)

1. **첫 번째**: 원래의 주문 파일(기존 주문변환과 동일한 성격의 파일)을 등록한다고 명시한다.
2. **두 번째**: 택배사에서 내려받은 **송장번호가 포함된 파일**을 등록한다고 명시한다.

---

## 3. 데이터 처리 흐름(요약)

```
[주문 파일] + [택배 송장 파일]
        ↓  (1차 병합: 내부 표준 헤더 체계)
   [내부 통합 테이블]
        ↓  (2차 병합·매핑: 사용자 업로드 양식)
   [쇼핑몰 업로드 형태의 결과]
        ↓
   미리보기 → 고정 입력 적용 → 다운로드
```

---

## 4. 1차 병합: 주문 파일 + 택배 송장 파일

### 4.1 기준 헤더(내부 표준)

- 서비스에서 사용 중인 **기준 헤더 전체(기존 29개 + 추가된 항목)**를 내부 캐논(canonical)으로 사용한다.
- 두 입력 파일의 열은 이 기준 헤더에 맞게 해석·정규화한 뒤 **하나의 통합 테이블**로 만든다.

### 4.2 조인(매칭) 방식

- 사용자에게 **조인 키 컬럼을 선택하게 하지 않는다.**
- **기준 헤더**를 기준으로 자동 병합한다. (구현 시: 동일 의미의 열 별칭·매핑 규칙은 기존 파이프라인의 기준 헤더·별칭 체계를 따른다.)
- 양쪽 파일에서 기대하는 키(예: 주문번호)가 기준 헤더에 매핑되지 않으면, **짧은 안내 메시지**로 실패 원인을 알린다. (별도의 다단계 선택 UI는 두지 않는다.)

### 4.3 1:N(한 주문에 송장 여러 개)

- 한 주문 건에 송장번호가 여러 개 매칭되면, **행을 복제**하여 내부 통합 테이블에 반영한다.
- 사용자는 이후 **미리보기**에서 동일 주문이 송장별로 여러 행으로 보이는 형태로 확인한다.

### 4.4 컬럼 충돌 시 우선순위(권장)

| 구분 | 우선 |
|------|------|
| 조인 키(주문번호 등) | **1번(주문 파일)** 기준 |
| 송장번호 및 택배사 측에서 오는 값 | **2번(택배 송장 파일)** 우선 |
| 그 외 동일 이름·동일 의미 열이 겹치는 경우 | **1번(주문 파일) 유지**를 기본으로 한다. |

업로드 영역에 **1번=주문, 2번=택배 송장**을 명시하여 정책과 UI가 일치하도록 한다.

### 4.5 누락

- 주문만 있고 송장이 없는 행: 송장 관련 칸은 비움(미리보기에서 빈 값으로 확인 가능).
- 송장만 있고 주문에 없는 행: 정책은 구현 단계에서 단일 규칙으로 고정(예: 미리보기 제외 또는 별도 요약). 본 문서에서는 **주문 중심 출력**을 기본 가정으로 둔다.

---

## 5. 2차 병합: 내부 통합 테이블 + 사용자 업로드 양식

### 5.1 입력

- 사용자가 등록한 **쇼핑몰 송장 업로드 양식 파일**(헤더 구조의 기준).

### 5.2 동작

- 4절에서 만든 **내부 통합 테이블**의 값을, 사용자 양식의 각 열에 매핑한다.
- 기존 주문·물류 변환에서의 **“업로드 양식과의 매핑”**과 동일한 개념으로 취급한다.

---

## 6. 미리보기 규칙

### 6.1 표시 기준

- 미리보기에 보이는 **열 이름·열 개수·열 순서**는 **사용자가 업로드한 양식 파일의 헤더**를 그대로 따른다.
- 내부 통합 테이블에 열이 20개 이상 있더라도, 사용자 양식이 **주문번호·송장번호 2열뿐**이면 **미리보기는 2열만** 표시한다.

### 6.2 의미

- 사용자에게 보여주는 것은 항상 **“쇼핑몰에 실제로 올릴 파일의 모습”**이다.
- 내부 표준 헤더로 가공된 중간 결과는 **UI에 전부 노출할 필요가 없다.**

### 6.3 고정 입력·다운로드

- 고정 입력으로 채울 값은 사용자 양식의 열 기준으로 적용한다.
- 다운로드 파일 역시 **사용자 양식 헤더 구조**를 유지한다.

## 7. 단계(스테이지)와 UI 대응

| 단계 | 내용 | UI 대응(권장) |
|------|------|----------------|
| 1 | 주문 파일 + 택배 송장 파일 업로드 및 1차 병합(내부 표준) | 상단 이중 파일 영역 |
| 2 | 쇼핑몰 송장 업로드 양식 등록 및 매핑 | 하단 카드 1(양식 등록) |
| 3 | 고정 입력 정보 | 하단 카드 2 |
| 4 | 최종 미리보기 확인 및 파일 다운로드 | 미리보기 + 하단 카드 3 |

## 8. 용어 정리

| 용어 | 설명 |
|------|------|
| 기준 헤더(내부) | 서비스 표준 필드 집합(29+α). 1차 병합의 기준. |
| 사용자 양식 헤더 | 쇼핑몰이 요구하는 업로드 파일의 열 구성. 미리보기·다운로드의 기준. |
| 1차 병합 결과 | 주문+택배 파일이 기준 헤더로 통합된 내부 테이블. |
| 2차 병합 결과 | 사용자 양식 헤더에 맞춘 최종 출력 직전 데이터. |
